What we learned about search — and the generators we built

We were hoping to teach an LLM to search these reasoning puzzles: enumerate candidate rules, test them against the examples, prune, lock. We didn't get there. This is a log of what we tried, what we observed (often a divergence from what we expected), and what we now speculate — plus the base generators we built along the way, shared in case they're useful to someone else.

▶ BIT viewer — GATE(f,g,h) calculator ▶ over-determination explorer ▶ mini-skill gallery ▶ generators (bit.py source)

Tone & scope. Nothing here is a confident claim or a leaderboard result. It's a set of things we tried and what we noticed — written in terms of what we hoped for and where reality diverged. We did not teach the model to solve gates; we're sharing the tools and observations in case they help. The generators/ show how to create each puzzle type (forward samplers; the solver/verifier and trace code we used to mint CoT on top of them were internal). Living page — last revised 2026-06-19; see the repo's git log for the exact commit.

In one paragraph

Just want to understand the BIT puzzle? Most people — and, it seemed, the model — never quite did. The BIT viewer explains it from scratch: it's one whole-byte gate over three streams; solving it means picking the simplest consistent rule (Occam on arity) — that matches the answer ~99% of the time, while the exact rule is uniquely determined only ~58% of the time. This section stands on its own.
What we built: a synthetic-data workflow for six puzzle families (sample a hidden rule → emit examples + a query → get the oracle answer → try many CoT trace formats, keep only steps a solver can derive), used to try teaching a 30B model to search these puzzles. It did not produce a reliable end-to-end BIT solver: BIT runs varied ~12–60%. We're releasing the generators, the interactive BIT views, a 5,891-row mini-skill sample, the BIT rule-space analysis, and the write-up below. The sections are independent — take any one and skip the rest.
One thing we saw (small evals): on two 64-row BIT evals the model picked the right answer from a given 8-candidate shortlist ~100% of the time, and from a shortlist it generated itself ~78–80%. On the 50–51 rows where its generated shortlist contained gold, its pick matched gold (50/50, 51/51); the wrong rows were rows where the shortlist did not contain gold. 50–51 rows is a small sample with run-to-run variance, so treat these as rough directions, not precise measurements. Row-level data: analysis/candidate_experiment_rows.jsonl.

Base generators as a place to try many CoT formats

What we tried: rather than commit to one trace format, in our working system we paired each forward generator with an exact solver/verifier, hoping the free oracle would let us cheaply try many custom CoT formats.

Scope note: the six generators released under generators/ are the forward samplers only (rule → examples → query → answer). The solvers and trace-validators we used internally are not part of this release; some raw trace-format code in EXAMPLE GENERATORS/ still imports modules we haven't published, so treat it as patterns to read rather than a runnable package.

Every generated row has a known latent rule and an exact gold answer (we sampled the rule), so the internal solver could check any intermediate state for free — though "known because we sampled it" is not the same as "uniquely identifiable from the displayed examples" (see BIT below). We hoped the free oracle would let us iterate formats quickly — per-bit column scan, whole-byte hypothesis-test, column-algebra search, state-trace enumerate→prune, compact one-liners, recovery/first-mistake, discriminate, DPO negatives — keeping only the ones whose every step is derivable. That part worked as we hoped: the generators made format experimentation cheap. Whether any format taught the model is a separate question (see the dissociation below).

Mini-skills, built from looking at the model's mistakes

What we tried: a library of 239 atomic "mini-skill" drills, each written after looking at how the model failed a puzzle — read the mistake, build a targeted drill for that gap.

When the model got a puzzle wrong, we looked at the failure and tried to distil the missing atomic step into a small self-contained drill (a single gate evaluation, a shift, a bijection check, a popcount-delta sign, …). The generators mint these on demand, so the library grew out of failure-watching rather than a fixed syllabus. We hoped the drills would shore up the gaps; we can't claim they moved the end result. The gallery shows up to two random generations per skill; the full random dump is miniskills_sample.jsonl (5,891 rows).

What "filtered" means (honestly): these are answer/format-consistency filtered, not "semantically coherent". We drop rows whose reasoning doesn't reach their boxed answer, remove exact-duplicate message pairs, and exclude three skills we found ambiguous — bit_scan_single (boxes an answer the reasoning never derives) and trans_style_audit / trans_style_pick (two style interpretations can render the same value, so "which style?" isn't always uniquely answerable). The filter is analysis/filter_miniskills.py; a fuller fix would add a "exactly one interpretation matches" guard in the generator itself.

Turning a task into micro-skill generators (the recipe)

The idea behind the mini-skills: the model could often do each step of a procedure on its own but not chain them (the alphabet point above). So we generated one drill per step. This is just task-decomposition turned into data — not magic — but it's a recipe you can reuse to build your own generators.
full task:   A ─────────────────────────▶ Z
                   decompose into the steps a solver actually does
             A ──▶ B ──▶ C ──▶ … ──▶ Z

per step, write a sampler that emits random instances of one step:
   "given A, produce B"     ┐
   "given B, produce C"     ├─  one micro-skill generator each
        …                   ┘   (mints instances on demand)
   "chain A ──▶ … ──▶ Z"    ───  plus one chaining drill

gate every emitted row on the solver  ──▶  keep only derivable ones
                                            = your mini-skill dataset

Concretely: list the mechanical steps your solver takes, write a sampler that emits random instances of one step at a time (and one that asks for the whole chain), and check each row against the solver so you only keep derivable ones. That's how the 239 drills behind miniskills.html were built; the recipe transfers to any task you can decompose into steps.

BIT as GATE(f, g, h) — a representation we found nice to look at

What we found helpful: reading the 8-bit output as one boolean gate over three whole-byte streams of the input, rather than eight independent per-bit problems. We offer it as a way to show and generate BIT, not as something we taught the model to solve.

A "stream" is the input run through one cheap op (shl, shr, rol, ror, complement, identity); the hidden rule is output = GATE(f(x), g(x), h(x)). On the real data the gate families we saw most were OR_XNOR and GATED_XNOR_NAND (an XNOR/NAND selector), with AND/OR/XOR close behind. In our hands, reading the byte whole covered more rows in generation and made the traces shorter than the per-bit view. One thing to watch when enumerating streams: for an 8-bit byte rol_k == ror_(8−k) — concretely rol1=ror7, rol2=ror6, rol3=ror5, rol4=ror4, rol5=ror3, rol6=ror2, rol7=ror1, and the same for their complements (~rol_k = ~ror_(8−k)). That collapses 14 of the 58 names onto 7+7 functions; the other 30 names (identity, complement, 14 shifts, 14 complemented shifts) are unique, leaving 44 distinct stream functions. We compared the function (truth table), not the label. One caveat worth stating: the sampler draws names uniformly, so an aliased rotation is sampled about 1.32× as often as a non-aliased stream — the prior is label-weighted, not function-uniform.

BIT as a search for FUNCTIONS vs a search for ANSWERS

Two ways to frame BIT: search for the function (recover the exact hidden rule) or search for the answer (just produce the query output). We spent most of our time on the first; Tong's per-column / frequency-prior approach looks like the second.

The whole-byte space is 1,379,240 syntactic rules, collapsing to 89,086 distinct functions; one (input → output) example leaves a median of ~700 of them. Within the whole-byte GATE(f,g,h) class — which reproduces 100% of the real train.csv BIT puzzles — here are the numbers, computed on those real puzzles (median 9 examples) and reproducible on random examples (analysis/measure_bit_space.py, analysis/occam_solver.py --stats):

QuantityValue
exact rule uniquely determined (function search)~58%
all consistent rules agree on the query answer~83%
a random consistent rule, applied to the query~93%
the simplest consistent rule (Occam on arity), applied to the query~99%
random over all functions, ignoring the examples0.39%

The rule is uniquely determined ~58% of the time; the answer is logically forced (all consistent rules agree) ~83%; the simplest consistent rule (lowest gate-arity + simplest streams) matches the gold answer ~99%. The generator builds each puzzle from a simple rule, so the simplest rule consistent with the examples is usually the one it used. On the real competition BIT rows our internal full solver scored ~99.3% (measured on our machine; train.csv isn't shipped here, so that figure isn't reproducible from this repo — the ~99% above is, on random examples via occam_solver.py --stats).

We spent most of our time on function search (enumerate→test→prune for the exact rule). Answer search predicts the output under a simplicity/frequency prior without insisting on a unique rule; Tong's per-column method (frequency priors + stride extrapolation) reads as that. The "Autotracer" (analysis/occam_solver.py) is the answer-search form: paste examples + a query, it returns the simplest consistent rule and the derivation.

One structural fact: 2,266 / 65,536 conceivable single (x→y) pairs are unreachable by any gate in this whole-byte family; an unreachable pair is inconsistent with the single-shared-gate model. In our corpus those were per-output-bit rows.

Encryption: a closed vocabulary seems to make it tractable

What we noticed: because every plaintext word is in a fixed list, an unknown cipher-word can be finished by testing every same-length vocabulary word and rejecting on the bijection.

Once enough letters are pinned by the examples, a partly-decoded word can be resolved by checking each same-length vocab word against the partial map and the one-to-one constraint (no two plaintext letters share a cipher letter). The closed vocabulary appears to be what keeps the type tractable and usually answer-determined rather than open-ended — though it's not guaranteed unique (a short query can occasionally admit more than one vocabulary-consistent reading).

Transformation — the one piece of ours that actually shipped

What actually shipped: we left the numeric transformation mostly unexplored — but the cipher (cryptarithm) transform rows from this same generator were the one part of our own work that made it into our best scoring blend.

The type has three independent knobs — operation × ordering × style — so the result layers an encoding on the raw arithmetic, and (unlike BIT) it can be genuinely under-determined from few examples. We didn't push numeric CoT-experimentation here. What we did end up using: about 2,500 verified cipher / cryptarithm rows (a CSP solve for the symbol→digit map plus honest priors), blended onto a strong public CoT base — and that blend was our best result. Honest caveat: we couldn't cleanly separate how much the cipher rows added versus the base, and cipher-type accuracy stayed low (~7.5%). But it is the part of our generator output that actually shipped, and it's where we'd take the platform next.

Numeric rate puzzles: clearing the decimals first (like a human)

What we tried: for d = ½·g·t² (and out = in·factor), cancelling the hidden decimal constant by ratio instead of recovering it — which kept those traces short in our generators.

The naive route recovers the rate by division (40.99 / 3² = 4.5544…) then multiplies it back — long decimal arithmetic we found error-prone to render. A human doesn't materialize g: since d ∝ t², take any example (t_e, d_e) and the query t_q and write d_q = d_e · (t_q / t_e)²; the unknown divides out. Same for unit conversion: out_q = out_e · (in_q / in_e). One honest caveat: the displayed example values are rounded, so this is a tolerance-compatible shortcut, not exact cancellation over the shown digits. In a quick simulation it reproduced the generator's exact 2-decimal answer about 74% of the time (gravitational) and 68% (unit conversion) from a single example, with nearly all predictions inside the 1% relative tolerance the task grades on. So it keeps traces tiny and stays within tolerance, rather than being provably exact.

Demo: generators/gravitational.py · generators/unit_conversion.py

Number conversion: no hidden parameter to infer

What we noticed: int↔Roman is a fixed, table-driven additive/subtractive notation (not a positional system) — a deterministic greedy walk of the value table, with nothing to deduce beyond the direction.

Int→Roman greedily subtracts/appends from the M, CM, D, … I table; Roman→Int scans left-to-right and subtracts when a smaller numeral precedes a larger one. The examples only signal which direction is being asked.

We came to see the CoT formats as search traces

What we hoped: most of our formats are search traces (enumerate → test → prune → lock), and we hoped our exact solver — a free step-verifier — would sidestep the weak/learned-verifier bottleneck the literature describes.

We started from a recursive bisection instinct: to get from a state A to the goal Z, if it fails in one shot, introduce a waypoint N, solve A→N and N→Z separately, and keep bisecting until each hop is trivial. That divide-and-conquer instinct led us to the search-supervision literature, whose training shapes are the same idea made trainable:

  • Teach-long enumerate→test→prune SFT — every step a solver-checkable state→action→observation→prune transition, rejected branches shown in-trace (contrastive CoT for free).
  • STaR / expert-iteration trim — inspired by Zelikman et al., STaR (2022) and Lehnert et al., Searchformer / "Beyond A*" (2024). To be clear about attribution: STaR samples and filters rationales (including rationalization conditioned on the answer); Searchformer learns explicit A* search dynamics and then bootstraps shorter successful traces. Our adaptation was the simpler "keep the shortest verified trace under a declining length budget, else fall back to the solver's canonical trace" — a synthesis, not either paper's method.
  • GRPO process / progress reward — GRPO is from Shao et al., DeepSeekMath (2024, arXiv:2402.03300). The specific reward shaping here — Hamming distance to gold and a "gold still reachable" progress signal for near-misses — is our proposed design, not something from that paper.
  • Step-level process supervision — Lightman et al., Let's Verify Step by Step (2023). Our solver emits the earliest-wrong-step label at zero cost.
  • Off the table at submission: tree-of-thought / MCTS / self-consistency — competition inference is greedy single-pass, so these stayed training-data tools only.

We had hoped a free exact verifier would be our edge where the field only approximates one. It didn't translate into a solving win for us — the next note is where things diverged.

Ranking given candidates vs generating them

What we did: a BIT answer is one byte (256 possibilities). We tried handing the model a shortlist of candidates (64 → 32 → 16 → 8) and having it pick; then we tried having it produce the shortlist itself. The pick accuracy was lower in the second case, on small evals.

Given a shortlist, the model picked the listed answer down to 8 candidates. When it produced the 8 candidates itself — from a deterministic recipe that derives them from the examples — end-to-end accuracy was ~78–80%. The breakdown of that condition on a 64-row eval (end-to-end = parse × candidate-recall × selection-given-present):

candidate set contained gold50–51/64 (~78–80%)
picked gold given its set contained gold50/50, 51/51

So on these rows, parsing held (64/64); on the 50–51 rows whose set contained gold, the pick matched gold; the wrong rows were rows where the generated set did not contain gold. Small sample (50–51 rows), so a direction, not a precise number.

So the errors we saw were in the generated set, not in parsing or selection. Small sample (50–51 rows), so this is a direction, not a precise measurement.

For context, from the BIT section: a random rule consistent with the examples matches the answer ~93% of the time, and the simplest consistent rule ~99%. The self-generated condition here was ~78–80%. Random over all functions (ignoring the examples) is 0.39%; within-slate baselines: a random 8-slate contains gold 3.1%, picking from a gold-containing 8-slate 12.5%.

Experimental detail. Runs A and B are two prompt variants (separate sampler checkpoints from the same BIT search lineage), each on the same 64-item eval; greedy decode (temp 0); the generated slate was required to be 8 candidates (it always was). Row-level evidence and an aggregation script that reproduces every cell above: analysis/candidate_experiment_rows.jsonl + analysis/aggregate_candidate_experiment.py.

These are our own internal evals (64-row sets) on the search-teaching BIT lineage specifically — not a leaderboard number. The raw end-to-end figures varied a lot across formats and runs (a worst of ~12% on a budget-fit deterministic search, up to ~50–60% in better runs); the 7/64–14/64 in this controlled condition is the deliberately-unscaffolded end of that range, and it's the quantified version of the "knows each step, can't recite the alphabet" gap.

How this compares to Tong Huikang's 0.85 (speculation)

We speculate: we used first divergence — the first example/position where candidate rules disagree — to build our atomic A→B drills, much like Tong's method. We suspect his approach worked because he put bit-teaching, search priors, and the answer into one trace, where we fragmented the same content across many micro-skills and formats. We did not verify this.

"First divergence" was the cheap, high-signal move for us: when several candidate rules survive, the informative thing is the first example (or bit position) where they split — show the candidate, show the position it first fails, reject. Our per-bit scans (i[0]=… no (pos 2)) and our discriminate / A→B drills are built on it. Tong Huikang's public 0.85 solution (github.com/tonghuikang/nemotron) uses a related per-column method with frequency priors and stride extrapolation.

Speculating from his public repo (we did not reverse-engineer it precisely): we suspect his published traces folded the bit computation, the search priors, and the answer into a single coherent end-to-end trace, and we wonder whether that "all in one trace" was the important difference. We instead split the same material across separate micro-skills and many formats, so the model may never have seen one trace teaching search, the bit op, and the answer together. If we tried again, consolidating into one unified trace is the first thing we'd test.

Budget & hindsight

What we'd change: the work became an exercise in trying to teach an LLM to search, and iterating that on a 30B model is slow. If we went again, we'd prototype the search-teaching loop on a much smaller model first, then try to transfer up. To be clear, that's about our own experimentation, not the submission — the competition target is fixed at Nemotron-3-Nano-30B-A3B; "smaller model" means a cheap prototyping rig to iterate faster, then transferring the lessons up to the fixed 30B, never submitting a different size.

We earmarked about $150 at the start — roughly $100 for Tinker (LoRA SFT / RL) and ~$50 / ~600 credits for Colab Pro+. We kept each experiment cheap — only about $2 of Tinker compute apiece — by running lots of small steps. To be precise about the bookkeeping: by "experiment" we mean one format/idea, which usually bundled several short training runs plus evals; the ~$100 spread over a few dozen such experiments, which is why the per-experiment figure and the larger run count below aren't in tension. Two things helped: exact, free oracles meant every generated trace was usable without paying for human labels or a learned verifier; and having CODEX (an agent) manage the Tinker runs end-to-end — launch a small step, evaluate, hand off, repeat, with no idle GPU or manual babysitting — saved us a lot of money.

Trace length was our other cost lever. Shorter chains-of-thought are cheaper both to train on and to run at inference (the competition caps output at 7,800 tokens), so for most of the competition we deliberately kept our traces tight — aiming under ~900 tokens, then relaxing to ~1.1k, then ~1.3k as harder cases needed more room. Only near the end did we spend the full token budget — letting the enumerate-test-prune search traces run long to try to teach BIT — which is also where our cost climbed. On the training side, we had good luck with QLoRA (4-bit LoRA) on an H100 (via Colab Pro+, alongside Tinker for the LoRA SFT/RL) — it worked especially well at the lower token counts, where short traces packed efficiently and runs stayed fast and cheap.

The scale that bought: roughly 950 Tinker runs (≈500 short training runs + ≈450 evaluations) across ~33 days, about 175 training-data builds, the BIT trace format alone revised through 30-plus numbered versions (up to V35), and 256 automated agent-to-agent handoff notes.

We had a strong early start — so we were hopeful, not going in blind. Before we leaned on a public CoT base, one of our own SFT-trace submissions already scored about 78% overall — which we believe placed us around top 50 on the leaderboard at the time — and notably that was with BIT solving only about 22%, so the five other puzzle types were already carrying the score. That early traction is why we stayed hopeful and kept pushing: the approach seemed to be working for most of the data, and BIT was the obvious holdout. Throughout, we kept doing holistic looks at the errors — reading what the model actually got wrong and distilling each gap into a targeted mini-skill — rather than guessing. BIT being the stubborn ~22% is exactly what pushed us into the search-teaching experiments below (which, on BIT specifically, varied widely — from a worst of ~12% up to ~50–60% in better runs; same hard type, harder framing).

Where the time went, and what our final submission was. Most of our effort went into trying to teach the model a different search method for BIT (the bisection / enumerate-test-prune work above), and that bulk never became a finished submission. Results varied a lot (these are rough recollections across a handful of runs, on small evals — so treat the percentages as ballpark, not careful measurements): our better BIT runs reached roughly 50–60%, an early attempt was ~22%, and the worst — a deterministic search that fit within budget — got only ~12%. In that ~12% case everything was locally solvable — like the alphabet: ask the model each step on its own ("what letter comes after A?", then "after B?") and it's right essentially every time; ask it to recite the whole alphabet and it drops to ~12%. It did each A→B step in isolation but did not chain them — the same pattern as the dissociation above (small samples, so a soft observation). (Our final leaderboard entry was small and almost incidental — we were just testing whether our cipher (cryptarithm) rows added any lift; those rows (see Transformation above) are the one part of our own work that shipped, blended onto a strong public CoT base — and by the time we looked up, we'd run out of time.)

Our read on where we iterated: most of what seems to matter for search-teaching — trace shape, length budget, verifier-gated trimming, where the model fakes a step — looks largely model-agnostic, so we suspect those lessons would have transferred up from a smaller model at a fraction of the cost. We can't prove it; it's the bet we'd make next time. The unexplored transformation lane and the STaR/GRPO loops are where we'd look first.

SPECULATION (hindsight-inspired) — an architecture hunch about MoE routing. Everything in this box is a hindsight-inspired hunch we could not confirm — us connecting our own routing struggles, after the fact, with a noise result we only really weighed later. Nemotron-3-Nano-30B-A3B is a mixture-of-experts: almost all of its capacity sits in the sparse experts, not the shared/attention layers. We reached about 0.76 relatively easily with LoRA on the shared layers only — no expert layers, which already gets you the bulk of the non-BIT types. Pushing further on BIT seemed to need the experts, and in hindsight we feel the hard part was getting expert routing to stay consistent across a long trace: a long enumerate-test-prune BIT trace only holds together if the same experts fire reliably step after step, so any routing drift reads as noise. We speculate — purely a guess — that some of that noise may even have been expert normalization. We're not the only ones poking at noise here: we recall a public gold-medal notebook reporting that simply adding noise improved their result — and perhaps that's why: if consistent expert routing is the real bottleneck, a little noise (or a different expert normalization) might be helping in a way none of us fully understands. We never pinned it down; we flag it in case it's a thread someone else can pull.
Why share something that hasn't paid off? It hasn't produced a reliable result for us so far — BIT runs ranged from ~12% to ~50–60% — and we're publishing it anyway. Whether that was a failure on our part or something more fundamental, we honestly don't know. We suspect the missing piece may be mundane (not enough training, too small a budget, or our cheap small-model experiments not transferring up to the fixed 30B target — the submission model size is the competition's, not something we'd change). Reading other teams' write-ups after the fact, the ones that got search-style traces to stick seem to have trained substantially longer / on more data than our short small-step runs, which makes us speculate we simply may not have trained long enough — though we can't separate that from format or model-size effects. So we've included lots of training traces (the 5,891-row mini-skill dump and the CoT formats) to give others ideas to build on — putting it in the open seems like the fastest way for someone to spot what we missed. And honestly, part of it is paying it forward: reading other people's public notebooks gave us insights along the way (the noise observation above is one), so contributing ours back is the least we can do. That's the point of an open contribution.
